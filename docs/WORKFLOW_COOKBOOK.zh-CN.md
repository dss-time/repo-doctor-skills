# 工作流实战手册

这份手册按“我要完成什么任务”组织，而不是要求你先记住所有 Skill 名称。每条工作流都说明调用顺序、选择原因、可复制提示词、权限变化、停止条件和安全交接方式。

若你只想查一个 Skill 的用途，请看 [Skill 完整目录](SKILL_CATALOG.zh-CN.md)；若你还没有完成安装，请先看[用户操作手册](USER_MANUAL.zh-CN.md)。

## 开始前：调用语法与安全约定

### 使用可移植的自然语言

本页提示词统一写成“请使用 skill-name……”，这样可以跨平台复制：

- 在支持已安装 Skill 的 Codex 中，可以把第一句改成 **$skill-name**，也可以通过 **/skills** 选择。
- ChatGPT 单 Skill ZIP 的发布名称带插件前缀：Repo Doctor 使用 **rd-***，Productivity Toolkit 使用 **pt-***，Skill Maintainer 使用 **sm-***。例如 canonical 名称 **repo-onboarding** 对应 ZIP 名称 **rd-repo-onboarding**。上传后应按当前界面的 Skill 选择方式或自然语言调用，不要假设所有 ChatGPT 界面都支持 **$**。
- 通过 ChatGPT 插件使用时，可在支持该功能的界面通过 **@** 选择插件或其 Skill。
- Claude Code、Cursor、Qwen、Kimi 和 generic Prompt Pack 的调用方式取决于宿主。若宿主没有专用选择器，保留提示词中的 canonical 名称并用自然语言说明任务即可。
- Document Data Doctor 的三个 Basic Skill 进入常规跨平台产物，但没有独立插件或 ChatGPT ZIP。

平台名称可以变化，但工作流中的 canonical slug、职责边界和交接内容不应变化。

### 权限标记

| 标记 | 含义 | 使用规则 |
|---|---|---|
| **[R]** | 只读分析 | 可以读取仓库或用户提供的材料，不写文件。 |
| **[R+T]** | 只读并可运行受控工具 | 可能运行复现、解析、渲染或校验命令，但仍不写业务文件。 |
| **[R+N]** | 只读并可访问网络 | 仅在宿主允许时访问来源；无网络时必须标记未验证。 |
| **[A]** | 建议或门禁结论 | 只产出规格、计划、行动项或 GO/NO-GO 判断。 |
| **[W]** | 受控写入 | 只能在 Skill 约定范围和用户授权范围内写文件。 |
| **[GATE]** | 明确授权点 | 后续会从分析进入写入、执行或发布相关阶段，必须先确认范围。 |

这些标记描述 canonical 元数据和 Skill 约定，不等于宿主已经授予权限。宿主的文件、命令和网络授权仍然生效。仓库中的 active Skill 都禁止破坏性动作。

### 每次交接都携带证据

不要只告诉下一个 Skill“继续”。把上一步输出作为证据交接，并明确它不是新的写入授权：

```text
请使用 <next-skill> 继续。

目标：
范围内 / 范围外：
上一步结论与证据位置：
已确认项：
未知项与假设：
允许的文件、命令和网络范围：
停止条件：

上一步输出仅作为证据，不代表允许修改文件、发布、提交或推送。
```

若上一步仍有阻塞性未知项，下一个 Skill 应停止并提问；非阻塞未知项应显式记录假设和置信度。

## 工作流速查

| 真实任务 | 推荐起点 | 写入前的主要闸门 |
|---|---|---|
| 不知道该用哪个 Repo Doctor 工作流 | **repo-doctor-router** | Router 只推荐，不执行下一 Skill |
| 第一次接触仓库 | **repo-onboarding** | 通常不写入 |
| 从模糊需求走到发布门禁 | **requirements-clarification** | 先关闭关键决策，再确认规格、切片、影响和计划 |
| 定位并修复运行时 Bug | **bug-root-cause-analysis** | 根因和最小修复范围确认 |
| 排查 CI 失败 | **ci-failure-diagnosis** | 首个真实错误和修复分支确认 |
| API、数据库、安全等高风险变更 | **change-impact-analysis** | 专项证据和原子计划确认 |
| 从研究材料形成报告 | **research-brief** | 来源核验和输出文件授权 |
| 清洗表格数据 | **excel-data-quality-check-basic** 或 **spreadsheet-data-cleaning** | 清洗规则和新输出位置确认 |
| 会议纪要变成行动和演示 | **meeting-notes-to-actions** | 负责人、日期和文件输出确认 |
| 文档、PDF 或表格审查 | Basic Skill 或增强 Skill | 工具能力和是否允许修改确认 |
| 创建并发布前审计 Skill | **skill-authoring** | 创建范围、生成物和发布操作确认 |

## 1. 第一次接触陌生仓库

**适合：** 你不知道技术栈、入口、构建命令、测试命令和阅读顺序。  
**流程：** **repo-onboarding [R]** → 可选 **project-health-check [R]**。

| 步骤 | 为什么用这个 Skill | 产出与权限 |
|---|---|---|
| 1. 仓库导览 | **repo-onboarding** 专注建立项目地图，不把“理解项目”扩大成体检或代码审查。 | 只读；输出技术栈、入口、命令、关键目录、阅读顺序和未知项。 |
| 2. 可选健康检查 | 只有在你还想知道测试、依赖、配置和维护风险时，才使用 **project-health-check**。 | 只读；输出广泛健康结论，不直接修复。 |

可复制提示词：

```text
请使用 repo-onboarding 理解当前仓库。

请基于实际文件说明技术栈、主要入口、构建命令、测试命令、关键目录之间的关系和建议阅读顺序。找不到证据的内容标记为未知，不要修改任何文件。
```

需要体检时再发起第二次调用：

```text
请使用 project-health-check 对刚才已理解的仓库做一次只读体检。

重点检查构建、测试、依赖、配置、文档和维护风险。请引用文件位置，并把事实、推断和未知项分开。不要实施修复。
```

**闸门与交接：** 这条流程默认始终只读。若结果建议修复，不要直接说“全部修掉”；应把具体问题转交给 **change-impact-analysis** 或 **safe-change-plan**，确认范围后再进入实现。

## 2. 从模糊需求走到发布门禁

**适合：** 用户提出了一个还不完整的功能需求，希望最终安全落地。  
**流程：** **requirements-clarification [R]** → **requirements-to-spec [R]** → 可选 **spec-to-work-items [A]** → **change-impact-analysis [R]** → **safe-change-plan [A]** → **[GATE]** → **safe-fix-implementation [W]** → **safe-code-review [R]** → **test-gap-analysis [R]** → 可选 **[GATE] safe-test-implementation [W]** → 可选 **documentation-sync [W]** → **release-readiness-check [A/GATE]**。

| 阶段 | 为什么这样排 | 关键边界 |
|---|---|---|
| 决策澄清 | **requirements-clarification** 读取仓库证据，只关闭会实质影响行为、兼容、安全、数据、迁移或验收的选择。 | 一次只问一个关键决策，在规格或代码前停止。 |
| 规格化 | **requirements-to-spec** 先把目标、范围、约束和验收标准说清楚。 | 不生成具体代码方案，不写文件。 |
| 交付切片 | 大规格使用 **spec-to-work-items** 创建垂直、可独立验证的工作项，标明依赖、冲突、测试和并行组。 | 不创建外部 Issue，也不替代代码级 `safe-change-plan`。 |
| 影响分析 | **change-impact-analysis** 用仓库证据识别受影响模块、接口、测试和风险。 | 不实施变更。 |
| 实施计划 | **safe-change-plan** 将已确认输入拆成原子步骤、验证和回滚。 | 需求仍模糊或影响未知时应停止并退回前一步。 |
| 实现 | **safe-fix-implementation** 只实施一个已确认问题或范围明确的修复。 | 写代码和运行命令前必须确认计划与文件范围。 |
| 审查与测试 | **safe-code-review** 审查变更；**test-gap-analysis** 分析缺口；只有用户同意后 **safe-test-implementation** 才写测试。 | 代码审查和缺口分析都不能被当成写入授权。 |
| 文档与门禁 | 用户可见行为变化时使用 **documentation-sync**；最后由 **release-readiness-check** 给出 GO、GO WITH CONDITIONS 或 NO-GO。 | 发布门禁不发布、不改版本、不打 tag、不提交、不推送。 |

第一阶段提示词：

```text
请使用 requirements-to-spec 把下面需求整理成可实施、可验证的规格：

<粘贴原始需求>

请从仓库证据消除疑问，给出范围内、范围外、约束、假设、待确认项和可测试的验收标准。只有阻塞性选择才提问，不要修改代码，也不要提前生成实施方案。
```

```text
请使用 change-impact-analysis 分析这份已确认规格的影响。

规格：
<粘贴规格或给出文件位置>

请列出直接和间接受影响的模块、接口、配置、测试、文档与兼容性风险，并给出证据位置。不要修改文件。
```

```text
请使用 safe-change-plan，根据已确认规格和影响分析生成原子实施计划。

每一步写明目标、文件或模块、修改内容、依赖、验证方法、失败回滚和停止条件。区分必须完成、建议完成和后续优化。不要执行计划。
```

**[GATE] 实现前确认：** 明确批准要修改的文件或模块、允许运行的验证命令、明确不修改的范围，以及何时停止。然后再调用：

```text
请使用 safe-fix-implementation，只执行已批准计划中的这一项：

<单一原子步骤>

允许修改：
允许运行：
明确不修改：
停止条件：

先保护工作区已有改动，完成最小变更后运行最相关验证。不要扩大范围，不要提交或推送。
```

实现后分开审查和补测：

```text
请使用 safe-code-review 只读审查当前变更。

请按严重度列出有证据的问题、影响和验证建议。不要自动修复，也不要把风格偏好标成高风险。
```

```text
请使用 test-gap-analysis，将已确认需求和当前 diff 映射到现有测试。

区分已覆盖、部分覆盖、缺失和无法判断的场景，按风险排序建议测试。不要创建测试文件。
```

若缺口值得补测，再单独授权：

```text
请使用 safe-test-implementation，只补充以下已确认的高价值测试：

<粘贴缺口和行为映射>

默认只修改测试、fixture 和必要的测试辅助代码。若必须修改生产代码，请停止说明原因。先跑最小相关测试，再跑合理范围回归。
```

文档与最终门禁：

```text
请使用 documentation-sync，根据已确认实现和 diff 只更新受影响文档。

保留现有语言、结构和术语；只写能够被代码或配置证实的内容。若发现实现错误，请报告阻塞，不要改业务代码。
```

```text
请使用 release-readiness-check 对当前候选版本做只读发布门禁。

请给出可复现证据、阻塞项、条件项、风险、回滚准备，以及 GO、GO WITH CONDITIONS 或 NO-GO 结论。不要发布、改版本、打 tag、提交或推送。
```

**安全交接：** 每一步只接收上一步的结论和证据；计划不是实现授权，测试建议不是测试写入授权，GO 结论也不是发布授权。

## 3. 定位并修复运行时 Bug

**适合：** 有稳定或间歇性的用户可见错误，需要先证明根因再修复。  
**流程：** **bug-root-cause-analysis [R+T]** → 可选 **safe-change-plan [A]** → **[GATE] safe-fix-implementation [W]** → **test-gap-analysis [R]** → 可选 **[GATE] safe-test-implementation [W]** → **safe-code-review [R]**。

| 步骤 | 为什么 | 停止条件 |
|---|---|---|
| 根因分析 | **bug-root-cause-analysis** 从第一个可信错误建立“输入/状态 → 路径 → 失效点 → 症状”的证据链。 | 无法复现、缺少关键日志或存在多个同等假设时，不得伪称根因已证实。 |
| 计划 | 风险较高、跨模块或回滚复杂时，用 **safe-change-plan** 将修复方向变成原子步骤。 | 输入不足时退回根因分析或提问。 |
| 修复 | **safe-fix-implementation** 执行经批准的最小修复。 | 出现范围外生产代码需求、测试显示不同根因或用户改动冲突时停止。 |
| 回归 | **test-gap-analysis** 先确定需要什么回归测试，**safe-test-implementation** 经授权后才写测试。 | 需要为了可测性改生产代码时先停止并申请扩大授权。 |

可复制提示词：

```text
请使用 bug-root-cause-analysis 分析这个 Bug：

症状：
影响：
环境：
复现步骤：
日志或错误：

从第一个可信错误开始，建立因果链。区分已证实根因、主要假设、替代假设和未知项；可以运行安全的最小复现实验，但不要修改文件或实施修复。
```

根因确认后：

```text
请使用 safe-change-plan，把已证实根因和最小修复方向整理成可验证、可回滚的计划。不要执行。
```

```text
请使用 safe-fix-implementation，只修复以下已证实根因：

根因与证据：
批准的文件范围：
验证命令：
停止条件：

做最小变更，不顺便重构。不要提交或推送。
```

```text
请使用 test-gap-analysis，为这个已确认 Bug 和修复分析回归测试缺口。先只给测试建议和真实运行命令，不要写测试。
```

**安全交接：** “可能原因”不能直接交给实现 Skill；至少要标明置信度和验证方式。若 CI 中才失败，先走下一条 CI 工作流，而不是强行当成本地运行时 Bug。

## 4. 排查 CI 失败

**适合：** workflow、job 或 step 在 CI 中失败，需要找到第一个真实失效原因。  
**流程：** **ci-failure-diagnosis [R]** → 按结论分支到 **safe-test-implementation [W]**、**safe-fix-implementation [W]** 或外部平台管理员 → 可选 **release-readiness-check [A/GATE]**。

| 诊断分类 | 推荐交接 |
|---|---|
| 测试本身错误、缺少回归或 flaky 测试 | 先确认修复范围，再交给 **safe-test-implementation**。 |
| 生产代码回归 | 如需影响分析或计划先补充，然后交给 **safe-fix-implementation**。 |
| 依赖或锁文件问题 | 先交给 **dependency-upgrade-analysis**；不要直接改锁文件。 |
| 权限、密钥或 Runner 环境 | 报告配置名称和位置，交给有权限的平台管理员；不要显示密钥值或降低安全控制。 |
| 独立于 CI 的运行时 Bug | 转交 **bug-root-cause-analysis** 建立运行时证据链。 |

诊断提示词：

```text
请使用 ci-failure-diagnosis 分析这次 CI 失败。

workflow / job / step：
CI 日志：
相关 commit 或 diff：
本地结果：

请定位第一个可信错误，区分代码、测试、依赖、缓存、权限/密钥、Runner 环境和 flaky failure。对比本地与 CI 的运行时、系统、环境变量名称、锁文件和命令；不要输出密钥值，也不要修改文件。
```

若确认是测试问题：

```text
请使用 safe-test-implementation，只修复诊断报告中已确认的测试问题。

允许修改的测试范围：
必须保持不变的生产代码：
最小复现命令：
回归命令：

若必须改生产代码，请停止。不要提交或推送。
```

若确认是代码问题：

```text
请使用 safe-fix-implementation，只修复 CI 诊断中已证实的生产代码问题。

证据：
批准的文件范围：
本地复现与验证命令：
停止条件：
```

**闸门与交接：** 没有完整失败 step、首个错误附近日志或 workflow 配置时，诊断应标记 UNKNOWN 并请求材料。CI 诊断只给方向，不自动修改工作流、权限或密钥。最后的发布就绪检查仍只是门禁，不会重跑或发布外部流水线。

## 5. API、数据库、安全等专项高风险变更

**适合：** 变更涉及兼容性、迁移、安全、性能、配置或架构决策。  
**流程：** **change-impact-analysis [R]** → 选择一个或少数相关专项 Skill **[R]** → 可选 **architecture-decision-record [W]** → **safe-change-plan [A]** → **[GATE] safe-fix-implementation [W]**。

先用下表选择专项 Skill，不要让一次调用吞掉所有审查：

| 风险问题 | 专项 Skill | 它不做什么 |
|---|---|---|
| 依赖版本、破坏性变更、许可证或生态兼容 | **dependency-upgrade-analysis** | 不修改依赖或锁文件。 |
| REST、GraphQL、RPC、事件或 SDK 契约 | **api-contract-review** | 不修改接口。 |
| schema、数据回填、锁表或零停机顺序 | **database-migration-review** | 不执行迁移，不连接生产库。 |
| 一个入口是否真的可以删除 | **dead-code-verification** | 证据不足时不建议删除，也不删除代码。 |
| 鉴权、输入、注入、SSRF、XSS、密钥等 | **security-focused-review** | 不自动攻击生产系统，不直接修复。 |
| 基线、profile、benchmark 或性能退化 | **performance-regression-analysis** | 不凭代码外观断言性能问题，不直接优化。 |
| 配置来源、优先级、环境漂移和危险默认值 | **configuration-audit** | 不显示密钥值，不修改配置。 |
| 需要记录真实的架构选择与权衡 | **architecture-decision-record** | 只写 ADR/架构文档，不虚构共识。 |

通用影响分析提示词：

```text
请使用 change-impact-analysis 分析这项高风险变更：

目标：
候选变更：
明确不变：

请先识别直接和间接影响、兼容性边界、数据/配置/接口风险、测试和回滚需求。不要实施变更。
```

然后只调用必要的专项。例如数据库迁移：

```text
请使用 database-migration-review 审查这份迁移方案和相关代码。

请检查 up/down、可逆性、数据丢失、默认值、空值、类型转换、索引、约束、锁表、回填、复制延迟和零停机部署顺序。给出证据、置信度、验证和回滚建议；不要执行迁移或连接生产数据库。
```

安全专项示例：

```text
请使用 security-focused-review 审查 <明确范围>。

先建立轻量威胁模型和信任边界。每条发现都给出证据位置、攻击前提、影响、可利用性、严重度、置信度和修复方向。不要执行攻击、访问生产系统、输出密钥或修改代码。
```

确实存在需要记录的决策时：

```text
请使用 architecture-decision-record，根据已有证据创建或更新 ADR。

问题：
候选方案：
已确认的决策人结论：
仍未决定的事项：

遵循仓库现有 ADR 目录和模板。未达成共识的状态必须写 Proposed；只允许修改 ADR 或架构文档。
```

**[GATE] 进入实现前：** 将通用影响和专项结论合并进 **safe-change-plan**，确认 API/数据库/配置兼容策略、部署顺序、回滚和停止条件。专项审查不是实现授权，ADR 也不是实施授权。只有明确批准的原子步骤才能交给 **safe-fix-implementation**。

## 6. 从研究问题到可交付报告

**适合：** 需要基于材料或网络来源完成研究简报，再形成报告或 PDF 审查结果。  
**流程：** **research-brief [R+N]** → 可选 **content-consistency-check [R+T]** → **[GATE] report-writer [W]** → 可选 **document-review [W]** → 若已有真实 PDF，再用 **pdf-review [R+T]**。

| 步骤 | 为什么 | 关键边界 |
|---|---|---|
| 研究简报 | **research-brief** 明确研究问题、证据标准、日期，并优先收集原始或官方来源。 | 无网络或资料时不得假装完成实时研究。 |
| 一致性核对 | 多份材料之间存在数字、日期、版本或结论差异时，用 **content-consistency-check**。 | 没有权威证据时不擅自选择“正确值”。 |
| 报告撰写 | **report-writer** 将已核验材料组织成面向读者的结构化报告。 | 写文件或导出前确认格式和输出位置；不补造事实。 |
| 文档审查 | **document-review** 检查结构、逻辑、事实、数字、引用和格式；默认给问题清单。 | 只有用户要求时才修改副本，并保留原意和结构。 |
| PDF 审查 | 只有真实 PDF 已存在且工具可用时，用 **pdf-review** 做文本、页码、表格和视觉检查。 | 无渲染/OCR 能力时必须说明限制，不能虚构页码或视觉验证。 |

研究提示词：

```text
请使用 research-brief 研究以下问题：

研究问题：
范围：
截止日期：
可接受来源：
排除范围：

优先使用原始、官方和高可信来源，记录发布日期、事件日期和访问日期。区分事实、推断、冲突信息和未知项。若无法联网或缺少资料，请明确降级，不要伪造来源。
```

有多份材料时：

```text
请使用 content-consistency-check 对照这些材料中的术语、数字、日期、单位、人物、版本、链接和结论。

请为每个冲突给出两侧位置、严重度、可能的口径差异和建议基准。没有权威证据时不要替我决定正确值。
```

**[GATE] 确认读者、结构、格式和输出位置后：**

```text
请使用 report-writer，根据已核验的研究简报和一致性报告撰写报告。

目标读者：
用途：
所需章节：
输出格式与位置：

区分事实、引用、推断、建议和未知；保持数字、日期、术语与引用一致。材料不足处写限制，不要自行补造。
```

审查和 PDF 交接：

```text
请使用 document-review 审查这份报告，先给必须修改、建议修改和可选润色清单。除非我再次明确授权，否则不要整篇重写或覆盖原文件。
```

```text
请使用 pdf-review 审查这个已生成的 PDF。

请说明文件是可搜索 PDF、扫描件还是 OCR 结果，尽量保留页码定位，并检查缺页、乱码、表格错位、数字矛盾、引用和版式。无法渲染或 OCR 时明确标记未验证。
```

**安全交接：** 来源清单必须随研究结果进入报告阶段。网络访问、文档写入和文件导出是不同权限；前一步能联网不代表后一步可以创建或覆盖文件。

## 7. 清洗表格但保留原始数据

**适合：** 想先了解数据质量，再按可追踪规则生成清洗后的新文件。  
**流程：** 只需审计时 **excel-data-quality-check-basic [R]**；需要清洗时 **spreadsheet-data-cleaning [W]** → 可选 **content-consistency-check [R+T]** → 可选 **report-writer [W]** 或 **presentation-outline [W]**。

| 选择 | 什么时候用 | 边界 |
|---|---|---|
| **excel-data-quality-check-basic** | 只需要只读地检查缺失、重复、类型、公式和异常。 | 不写文件、不运行脚本；不生成清洗结果。 |
| **spreadsheet-data-cleaning** | 用户明确要求清洗，并且平台具备表格处理能力。 | 原始文件保持只读；清洗结果另存；每条规则进入审计日志。 |
| **content-consistency-check** | 清洗结果还要和其他文档、报表或版本核对。 | 只报告冲突，不猜测权威值。 |

先做只读画像：

```text
请使用 excel-data-quality-check-basic 对 <文件> 做只读数据质量检查。

请报告工作表、列、类型、缺失、重复、异常、公式风险和无法判断项。不要修改原文件，也不要创建清洗副本。
```

若确定要清洗，先确认规则和输出位置：

```text
请使用 spreadsheet-data-cleaning 清洗 <原始文件>。

允许的清洗规则：
- <规则 1>
- <规则 2>

禁止猜测或填补的关键字段：
输出新文件：
审计日志：
需要保持的公式、行列和关键汇总：

原始文件必须保持不变。先给数据画像和规则预览；若规则会删除可能有效的数据、改变公式语义或覆盖原文件，请停止并征求确认。
```

清洗后验证：

```text
请对清洗结果和原始数据执行一致性核对。

请比较行列数、关键字段、日期/数字格式、公式、重复与缺失统计、关键汇总和审计日志。区分预期变化、意外变化和无法判断项。
```

**[GATE]：** 删除行、填补关键字段、修正规则性异常、改变公式、覆盖文件都必须单独确认。若表格工具不可用，Skill 应提供规则和限制，不得声称已生成或验证 XLSX/CSV。

## 8. 从会议记录到行动项和演示

**适合：** 把原始会议记录整理成决定、行动项和后续沟通材料。  
**流程：** **meeting-notes-to-actions [A]** → 人工确认负责人和日期 **[GATE]** → 选择 **report-writer [W]** 或 **presentation-outline [W]** → 可选 **content-consistency-check [R+T]**。

| 步骤 | 为什么 | 边界 |
|---|---|---|
| 结构化会议记录 | **meeting-notes-to-actions** 区分讨论、建议、决定和行动，合并重复项。 | 负责人或截止日期未明确时只能写 TBD，不能猜。 |
| 人工确认 | 会议参与者确认决定、Owner、日期和依赖。 | TBD 未解决时不得被下游当作承诺。 |
| 输出选择 | 需要正式记录时用 **report-writer**；需要讲述故事线时用 **presentation-outline**。 | 默认只生成大纲；PPTX 只有工具可用且用户明确要求时才生成。 |
| 一致性 | 对照会议记录、报告和演示中的数字、日期、名字与决定。 | 只报告冲突，不静默改写决定。 |

行动项提示词：

```text
请使用 meeting-notes-to-actions 整理下面的会议记录：

<粘贴会议记录>

输出摘要、明确决定、行动项、负责人、截止日期、依赖和待确认项。只提取记录中明确出现的信息；未知负责人或日期写 TBD。区分讨论、建议、决定和行动。
```

人工确认后，生成报告：

```text
请使用 report-writer，把已确认的会议决定和行动项整理成可分发的会议报告。

读者：
输出格式与位置：
已确认的 Owner 和日期：
仍为 TBD 的项目：

不得把讨论或建议改写成决定，也不得补造负责人或截止日期。
```

或生成演示大纲：

```text
请使用 presentation-outline，根据已确认的会议结果生成演示大纲。

受众：
目标：
场景与时长：
核心结论：

逐页给出目的、标题、要点、证据和建议视觉。缺少的数据标记占位和来源要求。默认只输出大纲，不生成 PPTX。
```

**安全交接：** 先把原始记录保存为证据，再保存结构化结果。人工确认是事实质量闸门；写报告或生成文件是权限闸门，两者不能合并。

## 9. Basic 与增强型文档、PDF、表格审查

**适合：** 你有 Word、PDF 或 Excel 文件，但不确定应选择低能力只读检查还是增强处理。  
**流程：** 先按任务意图和工具能力路由；不要因为工具不可用就假装完成增强审查。

### 路由决策

1. **只需基于可访问文本或结构做保守、只读检查？**
   - PDF：**pdf-review-basic [R]**
   - Word：**word-document-review-basic [R]**
   - Excel：**excel-data-quality-check-basic [R]**
2. **需要结构、逻辑、引用和格式审查，且可能在明确授权后修改文档副本？**
   - 使用 **document-review [W]**。
3. **需要 PDF 页码、表格、缺页、乱码、OCR 状态或视觉版式检查？**
   - 使用 **pdf-review [R+T]**；必须具备相应提取、渲染或 OCR 工具。
4. **需要实际清洗 Excel/CSV 并生成新文件？**
   - 使用 **spreadsheet-data-cleaning [W]**。
5. **需要跨多个文件核对术语、数字、日期、链接或结论？**
   - 使用 **content-consistency-check [R+T]**。

Basic PDF 示例：

```text
请使用 pdf-review-basic 对 <文件> 做只读基础审查。

只基于当前可可靠访问的文本和结构给出问题、证据和未知项。不要声称做过 OCR、页面渲染或视觉检查，也不要修改文件。
```

增强 PDF 示例：

```text
请使用 pdf-review 审查 <文件>。

我需要页码定位、表格和版式检查。请先确认是否能可靠提取文本、渲染页面和执行 OCR；能力缺失时明确降级范围。不要虚构页码、缺页或视觉结论。
```

Word 基础与增强：

```text
请使用 word-document-review-basic 对 <文件> 做只读基础审查，报告可证实的结构、术语、数字和明显问题。不要修改文件或声称验证了不可访问的格式。
```

```text
请使用 document-review 审查 <文件> 的结构、逻辑、完整性、术语、事实、数字、引用、语法和格式一致性。

默认只给问题清单和修改建议。若需要修改，请先说明可保持的格式能力、输出副本位置和无法验证的项目，未经授权不要覆盖原文件。
```

**[GATE]：** “只读”与“无需工具”不是同一个概念。工具不可用时应缩小结论或停止，而不是自动改用 Basic Skill 后仍声称完成 OCR、渲染、公式验证或格式保真。原始文件默认只读，任何转换或修订都写到新位置。

## 10. 创建 Skill 并做发布前质量审计

**适合：** 仓库维护者要新增一个值得复用的 Skill，并在发布前审计质量。  
**流程：** **skill-authoring [W]** → 仓库验证与构建 → **skill-quality-audit [R+T]** → 如获授权再回到 **skill-authoring [W]** 修正 → 单独的发布授权。

| 步骤 | 为什么 | 边界 |
|---|---|---|
| 适用性与创建 | **skill-authoring** 先判断需求是否值得成为 Skill，再检查重名、相邻 Skill、Pack 和公开边界。 | 一次只创建用户要求的范围；保护已有改动；不默认提交、推送或发布。 |
| 确定性验证 | 创建流程应运行仓库已有 validate、test、build，并检查生成物。 | 只修复当前 Skill 引入的问题，不顺便清理全仓库。 |
| 质量审计 | **skill-quality-audit** 检查结构、触发、工作流、资源、跨平台、安全和发布一致性。 | 审计严格只读；脚本只能判断确定性规则，语义质量由模型和 activation cases 判断。 |
| 修正循环 | 用户确认具体发现后，才回到写入类 Skill 做最小修正。 | P0/P1/P2 发现不是自动写入授权。 |

创建提示词：

```text
请使用 skill-authoring 在当前仓库中工程化创建一个 Skill。

真实使用场景：
2–5 个正向示例：
明确反例：
相邻 Skill：
目标 Pack / 插件：
公开或私有边界：
目标平台：

先判断它是否值得做成 Skill并检查重复与触发冲突，再规划最少必要的 SKILL.md、references、assets、scripts 和 UI 元数据。复用现有脚手架；不要创建空目录、Skill 内 README 或逐平台重复正文。默认不提交、不推送、不发布。
```

审计提示词：

```text
请使用 skill-quality-audit 对 <Skill、Pack 或仓库范围> 做发布前只读审计。

请检查结构与 frontmatter、触发质量、工作流和权限边界、渐进式加载、脚本测试、双语与跨平台、manifest/pack/生成物、安全和发布一致性。输出总体结论、评分卡、P0/P1/P2 发现、证据位置、影响、修复与验证建议。不要修改任何文件。
```

修正提示词：

```text
请使用 skill-authoring，只修复质量审计中以下已批准的问题：

批准的发现：
允许修改：
明确不修改：
需要重跑的验证：
停止条件：

不要处理未批准发现，不要改版本、提交、推送或发布。
```

**最终闸门：** validate、test、build 通过和质量审计给出可发布建议，仍不等于发布授权。版本修改、提交、push、marketplace 发布或对外分发必须作为新的、明确的用户请求。

## 11. 安全路由或交接工作

用户知道任务但不知道 Skill 时，使用 **repo-doctor-router [R]**。它返回已核验的下一 Skill、最短安全工作流、替代方案、必要输入和停止条件，但不执行推荐。上下文过长或负责人切换时使用 **session-handoff [R]**；交接只引用已有产物，区分事实与推断，清理敏感值，记录真实命令和结果，并给出可复制启动指令，不声称下一会话会自动读取。

## 通用失败处理

- **证据不足：** 标记 UNKNOWN，列出缺少的文件、日志、工具或选择；不要补造。
- **工具不可用：** 说明能够完成的降级范围和不能验证的内容；不要声称已运行、渲染、OCR、联网或生成文件。
- **权限变化：** 从 [R]、[R+T]、[R+N] 或 [A] 进入 [W] 前重新确认文件范围、命令和停止条件。
- **发现范围外问题：** 保存证据并报告，不顺便修复。
- **工作区已有改动：** 先识别并保护；无法安全隔离时停止。
- **验证失败：** 保留失败结果，回到产生问题的最小步骤；不得以“看起来没问题”代替通过。
- **发布相关请求：** **release-readiness-check** 只给门禁结论；实际发布、版本、tag、commit 和 push 始终需要独立授权。
